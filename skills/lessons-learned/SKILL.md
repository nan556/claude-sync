# Lessons Learned

## 调用时工作流程

1. **加载历史教训** — 读取 `~/.claude/skills/lessons-learned/lessons.json` 中的所有历史教训
2. **总结当前错误（如有）** — 询问用户本次项目中发生了什么错误或问题
3. **提炼教训** — 将错误转化为可操作的教训（不是列举错误，而是总结原则）
4. **保存新教训** — 将新的教训追加到 lessons.json
5. **展示完整教训列表** — 按分类展示所有教训

## 教训分类

- **编程习惯** — 语言特性、代码风格、开发工具相关
- **数据处理** — 数据来源、类型转换、命名准确性
- **业务逻辑** — 状态管理、UI 行为、用户交互
- **健康数据分析** — 数据核对、计算验证
- **根本原则** — 普适性原则，适用于所有项目

## 通用原则

### 1. 先理解根因，再动手
- 不要靠"试"来猜原因
- 代码行为有明确的原因，分析清楚后再改
- 如果不确定原因，先做假设验证，不要盲目修改

### 2. 用户明确说"不要这样做"时，立即停止并分析
- 用户说"不想重新初始化 BLE 适配器" → 分析为什么代码在尝试初始化、能否避免
- 用户的约束就是需求的一部分，不要绕过它去猜"更好"的方案

### 3. 动手前必须完整分析代码逻辑
- 用户说"你能不能跑一下代码查看完页面 自己梳理下逻辑" → 完整读取相关文件，分析调用链、数据流、状态变化
- 特别要分析：**两个操作之间有什么差异**，差异就是线索

### 4. 善用搜索
- 很多问题社区里已有解决方案，特别是微信小程序 BLE、uni-app、微信开发者工具等问题
- 先搜索再动手，不要闭门造车

### 5. 蓝牙/BLE 相关项目教训

**Android BLE 关键特性（已验证）：**
- `onBluetoothDeviceFound` 在 Android 上只报告**新的**蓝牙设备，曾通过 `createBLEConnection` 连接过的设备即使已断开，**也不再作为新设备报告**
- 解决方案：使用 `wx.getBluetoothDevices()` 获取系统缓存的设备列表
- `getConnectedBLEDevices` 只能获取当前真正建立了 GATT 连接的设备，无法获取缓存的已发现设备
- Android BLE 在有活跃 GATT 连接时，`startBluetoothDevicesDiscovery` 虽然返回成功，但 `onBluetoothDeviceFound` 可能无法发现新设备
- `closeBluetoothAdapter` 会关闭底层 BLE 适配器，**杀掉所有 GATT 连接**，不应该在需要保持连接的场景下调用
- `discovering: false` 不代表适配器正常，它只是一个状态标志

**蓝牙/BLE 项目的调试方法：**
- 观察日志中**第1次 vs 第2次的差异**，差异就是根因的线索
- 关键日志：适配器状态、搜索结果数量、设备发现回调
- Android BLE 问题用 `getBluetoothDevices()` 从系统缓存获取设备是最可靠的兜底方案

### 6. 生命周期相关教训
- `onMounted` 中使用 `await` 会阻塞生命周期，导致 `onUnmounted` 在页面还没完全挂载时就被触发
- `switchTab` 是同步的，`navigateToHomePage` 中的 `uni.switchTab` 调用后，后续代码可能还在执行时页面已经卸载了

## 历史教训（按分类）

### 编程习惯

**Lesson 1: 不要用 Python 思维写其他语言**
JS 的对象访问、Java 的类型系统各有各的规则，不能照搬。例如 JS 中没有 .get() 方法，re.sub() 的 flags 参数位置历史上有变化。

**Lesson 2: 避免重复声明变量**
用 linter 检查，ESLint 能自动发现重复声明问题。const activeSessions 重复声明导致页面什么都不显示。

### 数据处理

**Lesson 3: 序列化数据要确认类型**
从数据库/JSON 读来的字符串不一定是纯文本，可能是嵌套的序列化对象。从 SQLite 读取的 text 字段是 JSON 字符串，需要 json.loads() 解析。

**Lesson 4: 字段命名要准确**
monthly vs 5h 的差异会误导用户，命名要反映实际含义。5h 窗口不能叫 monthly。

**Lesson 5: 数据来源要可靠**
从 CLI 命令读取比读配置文件更不容易出错。MCP 服务器列表应该用 claude mcp list 而非读 mcp.json。

### 业务逻辑

**Lesson 6: 状态要有据可查**
默认折叠/展开、选择哪个会话等状态要有明确的业务规则。内存面板有内容时应该自动展开。

### 健康数据分析

**Lesson 7: 核对原始数据再计算**
心算容易出错，所有数据都要和原始来源逐一比对。25岁误当35岁导致最大心率全面错误。

**Lesson 8: 不要假设数据正确**
即使之前记录过，也要重新验证。静息心率 60 被误记为 70。

### 根本原则

**Lesson 9: 写代码前先理解数据流**
很多 bug 源于假设了错误的数据结构或来源。API 端点重构时响应过滤导致字段丢失。

**Lesson 10: 小错误会导致大后果**
一个 25/35 的心算失误让所有训练建议都错了。

## 案例

### 助听器小程序 BLE 双耳连接问题
- 症状：第1次进入扫描页能发现设备，第2次进入无法发现设备
- 错误做法：反复尝试重新初始化适配器、修改 `discovering` 状态
- 正确做法：
  1. 分析日志找出第1次 vs 第2次的差异
  2. 发现第2次进入时有 GATT 连接保持
  3. 搜索确认 Android BLE 限制：`onBluetoothDeviceFound` 不报告已连接过的设备
  4. 使用 `getBluetoothDevices()` 从缓存获取设备作为兜底
